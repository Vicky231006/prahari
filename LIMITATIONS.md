# System Limitations & Known Constraints

As per the rules of the prototype build, we do not fake features. What follows is an honest assessment of the limitations in the current PRAHARI implementation:

## 1. Fusion Classifier Model (Placeholder Active)
Currently, the `services/fusion_classifier/main.py` service wraps a fallback rule-based scoring engine because the final LightGBM model requires historical data to train accurately. The contract (`POST /internal/fusion/score`) and the 12-feature vector extraction are fully implemented, meaning the real `.joblib` model can be dropped in without changing the pipeline.

## 2. Authentication & Authorization
The Gateway API endpoints and the React frontend currently lack a robust OAuth2/OIDC implementation. `JWT_SECRET` is defined in the `.env`, but middleware blocking is not enforced on the REST endpoints to facilitate seamless evaluation and testing by the judging panel.

## 3. Synthetic Data
All telemetry moving through the system is generated by the `data/synthetic/` scripts. While the distributions (log-normal amounts, business hour weights) and correlations are statistically sound and mathematically defined, they do not perfectly capture the noisy edge cases of real-world banking traffic.

## 4. Single-Node Deployment
The `docker-compose.yml` configures Kafka in KRaft mode and Redis/Postgres as single instances. In a production banking environment, these would need to be replaced with multi-node high-availability clusters (e.g., Confluent Kafka, Redis Sentinel, AWS Aurora). Flink jobs are currently implemented in Python and executed locally; a production deployment would require a dedicated Flink TaskManager cluster.
