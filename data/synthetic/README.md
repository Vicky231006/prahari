# Synthetic Data — Generation Methodology

## Overview

All event data in PRAHARI is **synthetic**, generated by the scripts in `generators/`.
This is stated openly in the submission — no synthetic data is passed off as real.

## Event Types

### 1. `security-telemetry` (Kafka topic: `security-telemetry`)

| Field | Type | Description |
|---|---|---|
| event_id | UUID | Unique event identifier |
| identity_id | string | Bank identity (customer/employee) |
| timestamp | ISO-8601 | Event time (UTC) |
| event_type | enum | `login`, `privileged_cmd`, `endpoint_alert`, `geo_change` |
| source_ip | string | Client IP address |
| geo | object | `{lat, lon, country}` |
| device_fingerprint | string | Hashed device identifier |
| is_new_device | boolean | Whether this device is new for this identity |
| session_id | string | Session correlation ID |
| risk_flags | array | Detected risk indicators |

### 2. `transaction-events` (Kafka topic: `transaction-events`)

| Field | Type | Description |
|---|---|---|
| txn_id | UUID | Unique transaction identifier |
| identity_id | string | Bank identity |
| timestamp | ISO-8601 | Transaction time (UTC) |
| amount | float | Transaction amount in INR |
| currency | string | Always `INR` |
| channel | enum | `UPI`, `NEFT`, `RTGS`, `IMPS` |
| beneficiary_id | string | Recipient identifier |
| beneficiary_is_new | boolean | First-time beneficiary |
| session_id | string | Session correlation ID |
| is_cross_border | boolean | Cross-border flag |

### 3. `tls-handshake` (Kafka topic: `tls-handshake`)

| Field | Type | Description |
|---|---|---|
| session_id | string | Session identifier |
| timestamp | ISO-8601 | Handshake time (UTC) |
| key_exchange | enum | `RSA-2048`, `ECDHE-P256`, `ML-KEM-768`, `hybrid` |
| signature_algo | enum | `RSA`, `ECDSA`, `ML-DSA` |
| data_sensitivity | enum | `kyc`, `credit_history`, `routine` |
| bytes_transferred | integer | Bytes in session |
| destination | enum | `internal`, `external` |

## Generation Methodology

### Normal Traffic (~95%)

- **Identity pool**: 200 synthetic identities (`ID-00001` through `ID-00200`)
- **Transaction amounts**: Log-normal distribution (mean=8.5, sigma=1.5), yielding median ~₹4,900, 95th percentile ~₹150K
- **Timestamps**: Weighted 70% toward Indian business hours (09:00–18:00 IST)
- **Devices**: Each identity has 1–3 known devices; new device rate <2% in normal traffic
- **Beneficiaries**: Each identity has 2–8 known beneficiaries; new beneficiary rate <5%
- **Channels**: UPI 55%, NEFT 20%, IMPS 20%, RTGS 5% (mirrors real Indian payment mix)
- **Crypto**: 85% legacy (RSA-2048/ECDHE-P256), 15% PQC-ready (ML-KEM-768/hybrid)
- **Data sensitivity**: routine 85%, KYC 10%, credit_history 5%

### Injected Attack Scenarios (~5%, labeled)

Every injected event carries `scenario_type` and `is_synthetic_positive: true`.

#### 1. Account Takeover (ATO)
- New device + impossible travel (foreign geo) + high-value transfer (₹2L–₹10L) to a new beneficiary
- All events within a 15-minute window
- Risk flags: `impossible_travel`, `new_device`

#### 2. Insider Collusion
- Privileged account (first 50 identities) performs unusual data access (`privileged_cmd`)
- Within 10 minutes, a linked identity transacts to a shared/new beneficiary
- Risk flags: `unusual_data_access`, `privileged_account`

#### 3. Credential Stuffing → ATO
- Burst of 20 failed logins across multiple identities from 2–4 attacker IPs
- One successful login from the same IP cluster
- Immediate high-value transaction from the victim
- Risk flags: `failed_login`, `credential_stuffing`, `suspicious_ip_cluster`

#### 4. HNDL Exposure
- 3–5 TLS sessions carrying `kyc` or `credit_history` data
- Negotiated over legacy key exchange (`RSA-2048` or `ECDHE-P256`)
- Flagged per NIST FIPS 203/204 classification table

### Injection Rate

- One scenario injected every ~30 seconds (cycling through all four types)
- At ~23 normal events/second, this yields approximately 5% attack traffic

## Running the Generators

```bash
# Standalone (requires Kafka at localhost:9094)
python -m data.synthetic.generators.run_generators

# Via Docker Compose
docker compose up generators
```
